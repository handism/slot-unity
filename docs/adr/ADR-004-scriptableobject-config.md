# ADR-004: ゲームパラメータ管理に ScriptableObject を採用

**ステータス**: 承認済み
**日付**: 2026-03-20

---

## コンテキスト

シンボルの配当倍率・リールストリップ（出目テーブル）・ペイライン定義・ボーナス報酬範囲などのゲームパラメータをどこに持たせるかを決定する。
これらは調整頻度が高く、コードを変更せずに変えられることが理想。

## 選択肢

| 方式 | 概要 |
|------|------|
| **ScriptableObject** | Unity エディタ上で編集可能なデータアセット |
| コード内定数 | `const` / `static readonly` で直書き |
| JSON / CSV ファイル | Resources フォルダに配置して実行時読み込み |
| Excel / Google Sheets | 外部ツールで管理、インポートスクリプトで変換 |

## 決定

**ScriptableObject を採用する。**

対象アセット:
- `SymbolData` × 12（各シンボルの配当・スプライト・アニメ）
- `ReelStripData` × 5（各リールの出目テーブル）
- `PaylineData` × 1（25ラインの定義）
- `PayoutTableData` × 1（Scatter 配当、ボーナス報酬重み）

## 理由

- Unity エディタのインスペクタから直接値を変更・確認できるため、デザイナー（または今回は一人開発者）が素早くバランス調整できる
- コードコンパイルなしに変更・テストのサイクルが回せる
- アセット間参照（`SymbolData` → `Sprite` 参照など）が Unity の参照システムで管理されるため、パス管理が不要
- コードに定数を直書きするより、スクリプトとデータを明確に分離できる

## トレードオフ

- Git 管理において ScriptableObject（`.asset`）はバイナリ/YAML 形式のため、差分が読みにくい
  → 対策: `EditorSettings` で `Force Text` シリアライズを使用し YAML テキスト化する
- CSV / JSON より Unity 外部ツールから操作しにくい
  → 今回はチームが 1 人かつ Unity 前提のため問題なし
- 多量のデータ（1000 行以上のストリップ等）には不向き → 今回のリールストリップは 60〜100 マス程度のため問題なし
